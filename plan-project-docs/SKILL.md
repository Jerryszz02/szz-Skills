# Plan Project Docs

根据已完成的 plan 或现有项目的真实内容，只生成后续开发真正需要的项目指导文档。输出结果是可维护的 `docs/planning/` 目录，供之后的实现工作引用。

## 触发场景

- Plan 归档模式：Codex 为大型工程项目、主要功能、架构调整或产品方向产出正式 plan 后，用户要求保存计划、生成 PRD 类文档、创建项目指导文档，或在实现前准备参考文档。
- 现有项目梳理模式：用户针对一个已经存在且缺少相关项目文档的项目，明确提出“生成项目文档”“补一套项目文档”“根据现有项目生成 docs/planning”等类似请求。

## 工作流

1. 确认工作模式和来源。
   - 使用对话中最近的正式 plan；如果存在 `<proposed_plan>`，优先使用其中内容。
   - 如果多个 plan 互相冲突，使用用户最新认可或最新提供的版本。
   - 如果没有具体 plan，但用户明确要求为现有项目生成项目文档，进入现有项目梳理模式，不要要求用户补 plan。
   - 如果既没有具体 plan，也不是明确的现有项目文档生成请求，先要求用户提供 plan 或明确目标，不要自行编造范围。
2. 写入前先检查目标项目。
   - 从当前工作目录或用户明确给出的路径确认项目根目录。
   - 优先读取真实来源：`README*`、`AGENTS.md`、已有 `docs/`、包管理清单、构建清单、路由/API/schema 文件，以及可见的应用入口。
   - 使用只读命令做发现。不要只根据文件名推断 API、数据库表、部署目标或产品行为。
   - 在现有项目梳理模式下，先确认项目是否确实缺少覆盖当前项目的 `docs/planning/` 或同等规划文档；如果已有相关文档，优先更新索引或指出已有文档，而不是重复生成一套。
   - 在现有项目梳理模式下，必须记录实际检查过的文件或目录，并把所有结论绑定到这些证据；没有证据的功能、用户、架构、API、数据库、部署目标、时间线、负责人都写为 `待确认`。
3. 读取 `references/document-catalog.md`，选择最小文档集合。
   - Plan 归档模式下，只有当 plan 和仓库证据满足目录中的“需要生成”规则时，才生成对应文档。
   - 现有项目梳理模式下，只有当仓库证据满足目录中的“需要生成”规则时，才生成对应文档。
   - 先合并后拆分：如果两个文档服务同一批维护决策，优先合并到更核心的文档，并在 `README.md` 的跳过表中说明合并去向。
   - 默认最小集合是 `README.md` 加 1-3 篇真正需要长期引用的文档；只有复杂项目或重大变更才扩展到更多文件。
   - `project-brief.md`、`user-flow.md`、`architecture.md`、`decision-log.md` 容易和 `prd.md` 或 `technical-design.md` 重复，只有存在独立维护价值时才单独生成；否则把背景、流程、架构边界和关键决策合并到 `prd.md` 或 `technical-design.md`。
   - 没有 API 改动时不要生成 `api-design.md`；没有持久化改动时不要生成 `database-design.md`；没有部署/发布事项时不要生成 `release-plan.md`。
   - 对小型 bug fix 或窄范围重构，只生成最小有用的 planning note 和索引。
4. 创建或更新 `docs/planning/`。
   - 每次都创建或更新 `docs/planning/README.md` 作为索引。
   - 生成或更新任何 README 时，假设目标读者第一次接触该项目：必须说明项目是什么、为什么存在、由哪些部分组成、各部分如何协作、如何使用或继续开发、应该从哪些文件或命令开始。
   - 每个生成文件都要包含：文档目的、适用范围、来自 plan 或项目证据的具体内容、非目标，以及实现或验收指引。
   - 每个文档都从第一性原理写起：把前提、术语、输入、输出、依赖、约束、步骤、决策依据和验收方式写清楚；不要把任何细节当作读者已经知道。
   - 不要为目录清单中的每个类别填模板；如果文档正文主要是在重复另一个文档、`SKILL.md`、README 或现有代码，把它合并或跳过。
   - 如果目标文件已经存在，把新信息合并进去并保留人工写入内容。不要覆盖无关章节。
   - 对 plan 或项目证据中缺失的信息标记为 `待确认`；不要编造需求、schema、路由、凭据、时间线或负责人。
5. 汇报改动。
   - 列出已生成和已更新的 planning 文件。
   - 列出有意跳过的目录文档，并给出简短理由。
   - 提醒可能影响实现的未解决 `待确认` 项。
   - 在现有项目梳理模式下，必须附上人工检查建议：列出需要用户确认、修正或补充的地方，并说明它们为什么不能从当前项目证据中确定。

## 文件规则

- 默认只写入目标项目的 `docs/planning/` 目录，除非用户明确指定其他位置。
- 不要修改产品代码、测试、配置、secrets、`.env`、依赖清单或部署文件。
- 不要把 secrets、token、私钥、Cookie 值或环境变量值写入规划文档。
- 文档必须贴合当前 plan 或已检查到的项目证据，避免通用模板、臆测功能或宽泛架构文章。
- 在现有项目梳理模式下，文档必须明确写出“仅根据当前仓库可见内容整理；未在仓库中找到证据的内容均不做假设”或等价说明。
- 用户使用中文时默认输出中文；英文技术名、路径、API 和命令保持原样。

## README 生成要求

生成或更新 README 时，必须把它当作新读者的完整入口，而不是只做文件清单。

- 假设目标读者对项目一无所知，从零开始解释项目的目标、适用场景、核心概念、目录结构、关键文件、运行或使用方式、常见维护入口，以及后续开发应该阅读哪些文档。
- README 读完后，读者应能理解这个项目如何运作、主要流程如何串联、自己要如何使用或维护它，以及哪些信息仍然是 `待确认`。
- 不要用“显然”“按常规”“同上”“略”等方式跳过关键背景；如果某个细节无法从 plan 或项目证据确认，写成 `待确认`，并说明缺少什么证据。
- README 可以保持简洁，但不能依赖隐含上下文；必要时用表格、步骤或小节把信息拆开，让零基础读者能按顺序阅读。

## 细节和假设要求

- 任何文档都不要默认或假设未被 plan、仓库证据或用户明确说明的信息。
- 从第一性原理出发写清楚：问题是什么、为什么需要该方案、涉及哪些对象、对象之间如何交互、输入输出是什么、执行步骤是什么、成功如何判断、失败或未知项如何处理。
- 对项目中已经存在但名称不自解释的文件、命令、配置、角色、数据流或工作流，必须补上上下文和用途；不要只列名称。
- 对缺失证据使用 `待确认`，并尽量写明需要用户确认的具体问题或需要补充的具体文件、配置、命令输出。

## 索引要求

`docs/planning/README.md` 必须包含：

- plan 标题、项目文档生成请求或简短描述；
- 生成或更新时间；
- 已检查的项目根目录；
- 已检查的关键项目证据；
- 已生成或已更新文档表格，并为每个文档写一句用途；
- 已跳过目录文档表格，并写明跳过原因；
- 未解决问题或 `待确认` 项，如果有。

## 单篇文档要求

每个生成的 planning 文件默认包含以下章节；如果某类文档明显更适合其他结构，可以调整：

```markdown
# 文档标题

## 文档目的

## 适用范围

## Plan 或项目证据

## 非目标

## 实现指引

## 验收标准

## 待确认
```

使用简洁正文；当表格能让实现决策更容易扫描时，优先使用表格。只有当某个章节确实不适用于该文件时，才删除空章节。

## 资源

- `references/document-catalog.md`：文档选择规则、适用范围定义、必需内容和跳过条件。
