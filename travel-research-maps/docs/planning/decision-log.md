# travel-research-maps 决策记录

## 文档目的

记录 Firecrawl 多平台来源扩展后的关键产品、架构和安全取舍，减少后续维护时重复争论同一问题。

## 适用范围

适用于 `travel-research-maps` skill 的旅行研究流程、评分实现和地图写入边界。本文档仅根据当前仓库可见内容整理；未在仓库中找到证据的内容均不做假设。

## Plan 或项目证据

| 证据 | 决策主题 |
| --- | --- |
| `SKILL.md` | Firecrawl 是主研究路径；不输出日程式行程；默认输出审核清单。 |
| `SKILL.md` | 覆盖门槛不足时停止后续流程。 |
| `score_candidates.py` | 评分脚本只做确定性本地评分，不做网络请求。 |
| `scoring-rubric.md` | 营销内容完全排除，严重风险直接排除，优先项需要多平台正向证据。 |
| `README.md` | Google Maps 写入必须明确批准。 |

## 决策

| 日期 | 结论 | 理由 | 后果 |
| --- | --- | --- | --- |
| 2026-06-29 | 默认交付审核清单，不交付按天行程 | 当前 skill 目标是地点研究和地图收藏，不是路线规划。 | 输出更可审计，但用户若要行程仍需另行请求或新增 skill。 |
| 2026-06-29 | Firecrawl MCP 作为主研究路径 | 旧 Chrome 小红书路径依赖账号状态和页面可访问性，稳定性不足。 | 研究覆盖转向公开可抓取内容；平台限制会以缺口报告呈现。 |
| 2026-06-29 | 按目的地选择平台优先级 | 中国大陆和境外目的地的高价值社媒来源不同。 | 中国大陆以小红书/Bilibili 为主，境外以小红书/YouTube/Instagram/公开旅行内容为主。 |
| 2026-06-29 | Google Maps 只做核验和批准后保存 | Google Maps 更适合地址、营业状态和关闭风险核验。 | 评论不直接计入推荐分，但可形成 `hard_risks`。 |
| 2026-06-29 | 优先项需要至少 2 个正向平台 | 降低单一平台热点、搬运或营销偏差。 | 单平台高分候选会降为备选/待确认。 |
| 2026-06-29 | 评分器保持离线确定性 | 证据收集和事实核验由 workflow 处理，脚本只负责评分。 | 单元测试简单稳定；脚本不能自行补全事实或抓取网络。 |
| 2026-06-29 | 覆盖门槛不足时停止，不输出部分地点清单 | 防止用户把不完整研究误解为完整推荐。 | 失败时交付的是缺口报告，而不是较少数量的推荐。 |
| 2026-06-29 | Google Maps 写入必须等待人工批准 | 写入列表是外部副作用，且使用用户账号会话。 | 默认流程安全，但需要一次显式确认步骤。 |

## 被考虑或排除的选项

| 选项 | 状态 | 原因 |
| --- | --- | --- |
| 继续要求 Chrome 已登录小红书 | 排除 | 稳定性不足，且与 Firecrawl 多平台目标冲突。 |
| Firecrawl 不可用时自动退回内置 web search | 排除 | 会改变证据来源边界和可审计性。 |
| 自动绕过验证码或登录墙 | 排除 | 违反安全边界和平台访问控制。 |
| 评分器直接联网抓取事实 | 排除 | 会破坏离线确定性和可测试性。 |
| 未达到覆盖门槛时输出部分清单 | 排除 | `SKILL.md` 要求停止后续流程。 |

## 实现指引

- 修改上述决策时，应同步更新对应的 `SKILL.md`、评分规则、测试和相关 planning 文档。
- 如果新增行程规划能力，应该明确它是本 skill 的扩展还是另一个 skill。
- 如果引入新的来源平台，必须保留作者或频道、日期、立场、营销识别和独立性去重要求。

## 验收标准

- 后续改动不会无意绕开人工批准、覆盖门槛、多平台优先门槛或离线评分边界。
- 新决策有明确证据和影响说明，而不是散落在实现细节中。
